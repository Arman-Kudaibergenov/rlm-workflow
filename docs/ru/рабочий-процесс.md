# Рабочий процесс: три ритуала

Три кодовые фразы управляют жизненным циклом сессии. Достаточно написать одно слово — Claude знает что делать.

---

## `контекст` — начало сессии

Запустите в начале любой рабочей сессии.

```
контекст
```

**Что происходит:**

1. `rlm_start_session(restore=true)` — восстановить предыдущее состояние
2. `rlm_enterprise_context(query="текущий проект и статус работы")` — загрузить контекст проекта
3. `rlm_search_facts(query="PENDING tasks next session", keyword_weight=0.8, top_k=15)` — найти незавершённые задачи
4. Фильтрация: показывать только задачи текущего проекта (подавлять чужие `[project: other]`)
5. Если PENDING не пустой → объявить первую задачу и сразу начать
6. Если PENDING пустой → спросить что делаем

**Результат:** Claude знает где вы остановились, что ещё не сделано, какие решения были приняты. Никаких повторных объяснений.

## `суммаризируем` — конец сессии

Запустите когда заканчиваете работу или когда контекст приближается к 60–65%.

```
суммаризируем
```

**Что происходит:**

1. Прочитать буфер авто-захвата (`~/.claude/autocapture-buffer.jsonl`) — список изменённых файлов
2. Сохранить в RLM:
   - Завершённые задачи и изменения файлов
   - Архитектурные решения и их причины → `rlm_record_causal_decision`
   - Технические детали: IP, порты, конфиги, найденные ошибки и решения → `rlm_add_hierarchical_fact`
3. Обязательно сохранить PENDING-факт:
   ```
   PENDING tasks next session [task_id: project-feature-2026-02-28]:
   1) Дописать тесты для модуля оплаты.
   2) Code review от коллеги.
   ```
4. `rlm_sync_state()` — зафиксировать состояние
5. `git status` → закоммитить завершённое / добавить WIP в PENDING
6. Очистить буфер: `echo "" > ~/.claude/autocapture-buffer.jsonl`
7. Сказать пользователю запустить `/clear`

**Правила качества PENDING:**
- Включать ТОЛЬКО задачи которые точно не завершены
- Неизвестный статус → `❓`: `❓ загрузка в БД — не было явного подтверждения`
- Задачи другого проекта → отдельный факт с `[project: имя-проекта]`
- Если нет незавершённых → `PENDING tasks next session: none`

## `новая задача` — начало новой задачи

Когда начинаете работу над новой задачей (не продолжение старой).

```
новая задача
```

**Что происходит:**

1. `rlm_start_session(restore=false)` — чистая сессия для новой задачи
2. Очистить буфер авто-захвата
3. Оценить: нужен ли brainstorm? (новая фича → да; баг в 1–2 файлах → нет)
4. Сгенерировать `task_id` — человекочитаемый slug: `проект-фича-2026-02-28`
5. Сохранить TASK START в RLM:
   ```
   TASK START [project-feature-2026-02-28]: <описание задачи>.
   Подход: solo/team. Ожидаемые файлы: <список>.
   ```
6. Для сложных задач: создать команду агентов, назначить задачи, запустить

## Типичный рабочий день

```
Понедельник, 9:00
Вы:   контекст
AI:   [восстанавливает контекст]
      Незавершённые: 1) Тесты для планировщика. 2) Code review.
      Начинаю с первой задачи...

[работа, 2 часа]

Контекст заполнился до 68% → auto-summarize hook сработал автоматически
AI:   [сохранил в RLM, спрашивает запустить /clear]

Вы:   /clear

Понедельник, 11:30
Вы:   контекст
AI:   [восстанавливает]
      Незавершённые: 1) Code review (тесты закончены). 2) ...
      Начинаю с code review...
```

Никаких потерь контекста между очистками. Непрерывная работа через дни и недели.

## task_id — сквозной ключ задачи

Каждая задача получает уникальный slug-идентификатор:

```
project-feature-YYYY-MM-DD
```

Примеры:
- `forte-payment-schedule-2026-02-28`
- `infra-ollama-gpu-2026-01-15`

Этот ключ вставляется во все RLM-факты задачи:
- `TASK START [task_id]` — базовый факт при старте
- `TASK PROGRESS [task_id]` — промежуточные сохранения
- `PENDING tasks next session [task_id: ...]` — связь для восстановления

При ретроспективе (`итоги проекта`) — все факты задачи собираются по этому ключу из любого количества сессий.

---

## Принципы качества PENDING

| Правило | Пример |
|---------|--------|
| Только незавершённые | ~~"написать тесты (сделано)"~~ → не включать |
| Достаточно контекста | "дописать тесты для `payment/scheduler.py`" вместо "тесты" |
| Неизвестный статус | `❓ загрузка в БД — не было подтверждения` |
| Другой проект | отдельный факт `[project: other-project]` |
| Пусто → явно | `PENDING tasks next session: none` |
