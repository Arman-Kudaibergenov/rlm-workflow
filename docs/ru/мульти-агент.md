# Мульти-агентный workflow

Уникальное расширение этого workflow: RLM как **общая память команды агентов**.

## Проблема

При запуске нескольких агентов Claude для сложной задачи каждый имеет изолированный контекст.
Агенты общаются через `SendMessage`, но это эфемерно — пропадает при рестарте.

RLM обеспечивает персистентный слой памяти, доступный всем агентам.

## Архитектура

```
Лидер (Opus)
├── Сохраняет контекст задачи в RLM перед запуском команды
├── Создаёт задачи через TaskCreate
├── Запускает агентов через Agent tool с team_name
└── Получает результаты через SendMessage

Агент A (Sonnet)                    Агент B (Sonnet)
├── rlm_start_session(restore=true) ├── rlm_start_session(restore=true)
├── Читает контекст задачи          ├── Читает контекст задачи
├── Работает над своими задачами    ├── Работает над своими задачами
├── Checkpoint в RLM после каждой  ├── Checkpoint в RLM после каждой
└── Отчитывается через SendMessage  └── Отчитывается через SendMessage
```

## Важное ограничение

Каждый `rlm_start_session()` создаёт **изолированную сессию**. Факты, записанные агентом A, НЕ видны лидеру через `rlm_search_facts`. Данные пишутся, но сегментированы по session_id.

**Решение:** принять это ограничение как данность.

**Правило:** критические результаты обязательно передавать через `SendMessage`. RLM-чекпойнты агентов — страховка при сбое, не основной канал.

## Обязательный блок в промпте агента

Каждый агент при запуске должен получить этот блок дословно:

```
MANDATORY RLM RULES (в субагентах нет хуков — делайте это вручную):
1. СТАРТ: вызвать rlm_start_session(restore=true) перед любой работой.
2. ПОСЛЕ КАЖДОЙ ЗАДАЧИ: вызвать rlm_add_hierarchical_fact(
     content="TEAM [team-name] checkpoint: [ваша роль] завершил [название задачи].
      Файлы: [список изменённых файлов]. Решения: [ключевые решения]. Далее: [что осталось].",
     domain="team", level=1)
3. ПЕРЕД тяжёлым чтением (10+ файлов): сначала сохранить текущие находки в RLM.
4. ПРИ shutdown_request: сохранить финальное состояние в RLM, затем подтвердить shutdown.
Пропуск чекпойнтов = потеря работы при переполнении контекста.
```

## Управление здоровьем контекста агентов

У агентов нет хуков (context-monitor, pre-compact). Лидер отвечает за их состояние.

**Паттерн ротации:** для задач с 5+ подзадачами или тяжёлым чтением файлов — лидер завершает агента после 2–3 выполненных задач, запускает свежего. Новый агент читает контекст из RLM перед стартом.

**Мониторинг лидером:** если ответы агента становятся менее связными или повторяются — контекст переполнен. Немедленный checkpoint + ротация.

**Каскад при CRITICAL у лидера (автоматически через context-monitor.ps1):**
1. Broadcast checkpoint всем агентам
2. Ждать их RLM-сохранений (или 30 секунд)
3. Shutdown всех агентов
4. Сохранить состояние самому
5. Сказать пользователю `/clear`

## Формат checkpoint агента

```
TEAM [team-name] checkpoint: [роль] завершил [задача].
Файлы: [path1, path2]. Решения: [ключевые решения]. Далее: [что осталось].
```

Domain: `team`, Level: 1.

## Пример: рефакторинг в две руки

```
# Лидер запускает команду
1. Создаёт задачи: "рефакторинг модуля А" + "рефакторинг модуля Б"
2. Сохраняет в RLM: архитектурные ограничения, какие файлы трогать нельзя
3. Запускает Агента A (модуль А) + Агента Б (модуль Б)
4. Агенты читают ограничения из RLM, работают параллельно
5. После каждой задачи — checkpoint в RLM
6. Финальные результаты — через SendMessage лидеру
7. Лидер делает code review, отдаёт замечания через SendMessage
8. Shutdown команды, TeamDelete, итоговый commit
```
