# Что такое RLM и зачем он нужен

## Проблема: контекст не помнит ничего

Claude Code — мощный инструмент, но у него есть фундаментальное ограничение: **контекстное окно**.

Каждая сессия начинается с нуля. Когда контекст заполняется и срабатывает `/compact` или перезапуск — всё исчезает:

- Почему выбрана именно эта архитектура?
- Какие задачи были в процессе?
- Что не работало и как это починили?
- Какой формат данных у внешнего API?

Следующая сессия — Claude снова задаёт те же вопросы, снова изучает тот же код, снова совершает те же ошибки.

## Почему стандартные подходы не работают

| Подход | Проблема |
|--------|----------|
| `CLAUDE.md` в проекте | Не масштабируется: 200 строк — предел до загромождения |
| Summarize в `/compact` | Теряет детали, нет поиска, нет истории решений |
| Перечитывать историю чата | Невозможно — чаты не сохраняются между сессиями |
| RAG по кодовой базе | Ищет код, но не решения и не контекст разговоров |

## Что такое RLM

**RLM (Recursive Language Model Toolkit)** — это внешний MCP-сервер, который даёт Claude персистентную память.

Вместо того чтобы пытаться уместить всё в контекстное окно, RLM хранит факты, решения и прогресс во внешней базе. Claude обращается к этой памяти через MCP-инструменты.

```
┌─────────────────────────────────────────────────────────────┐
│                     Ваша сессия Claude                      │
│                                                             │
│  контекстное окно  ←──── rlm_enterprise_context() ────────┐ │
│  (200k токенов)           (только нужное, <3k токенов)     │ │
│                                                             │ │
│  rlm_add_hierarchical_fact() ─────────────────────────────┘ │
│  rlm_record_causal_decision()                               │
│  rlm_sync_state()                                           │
└──────────────────────────────┬──────────────────────────────┘
                               │ MCP (HTTP)
                               ↓
┌─────────────────────────────────────────────────────────────┐
│                      RLM-сервер                             │
│                                                             │
│  Иерархические факты (L0–L3)                               │
│  Причинно-следственные решения                             │
│  Семантический + keyword поиск                             │
│  Персистентное хранилище (SQLite / векторная БД)           │
└─────────────────────────────────────────────────────────────┘
```

## Ключевые концепции

### Иерархические факты (L0–L3)

Факты хранятся по уровням важности:
- **L0** — критические, глобальные (редко меняются)
- **L1** — рабочие факты сессии (задачи, решения, прогресс)
- **L2** — детали реализации
- **L3** — временные заметки

### Причинно-следственные решения

Не просто "что сделано", а "почему так сделано":

```python
rlm_record_causal_decision(
    decision="Использовать SQLite вместо PostgreSQL",
    reasons=["Один сервер, нет смысла в сетевой БД"],
    consequences=["Проще деплой", "Нет репликации"],
    alternatives=["PostgreSQL — избыточно для MVP"]
)
```

### Семантический + keyword поиск

RLM ищет факты тремя способами одновременно:
- **Semantic** — по смыслу (векторное сходство)
- **Keyword** — по точным словам (BM25)
- **Recency** — по свежести

Веса настраиваются под задачу. Для поиска PENDING-задач используем `keyword_weight=0.8` — точная строка важнее семантики.

## Что мы добавили к оригиналу

RLM-Toolkit создан для обработки больших контекстов (10M+ токенов). Мы адаптировали его под **повседневную разработку с Claude Code**:

1. **Ритуалы сессии** — три кодовые фразы управляют жизненным циклом: `контекст`, `суммаризируем`, `новая задача`
2. **Автоматизация через хуки** — память сохраняется без участия пользователя
3. **Мульти-агентная память** — несколько агентов Claude используют общее хранилище
4. **PENDING-паттерн** — структурированные списки незавершённых задач с привязкой к проекту
